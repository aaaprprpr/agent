当前已经能够支持fastapi远程调用模型，可以在本地开发，对b4进行非侵入适配












你可以把这个项目理解成“命令行驱动的 Agent 演示框架”，不是一个交互式聊天 App。

整体运行流是：

```text
runtime_input.json
  -> B1 总控
  -> 读 prompts 系统提示词
  -> 读 memory 记忆
  -> 调 B4 模型
  -> 模型决定是否调工具
  -> B3 执行工具，工具从 data 里读文件/表格
  -> B4 再根据工具结果回答
  -> 输出到 outputs
  -> 可选写回 memory
```

**data 是啥**
`data/` 是“输入样例 + 被工具读取的本地资料”。

里面几类东西：

- `data/runtime_input.json`：完整 Agent 的任务入口，用户输入就在这里。
- `data/runtime_input_*.json`：其他任务样例，比如计算、搜索、表格分析、格式转换。
- `data/docs/`：给 `file_reader`、`local_file_search` 这些工具读取的本地文档。
- `data/tables/`：给 `table_analyzer` 读取的 CSV/TSV。
- `data/tool_inputs/`：单独测试 B2 skill 时用的输入。
- `data/messages/`：单独测试 B3/B4 时用的消息样例。
- `data/memory_inputs/`：单独测试 B5 保存记忆时用的样例。

所以 `data` 不是记忆库，它更像“任务输入和演示素材”。

**memory 是啥**
`memory/` 是长期记忆存储。

- `memory/memory_index.json`：记忆索引。
- `memory/global/`：全局记忆。
- `memory/conversations/`：历史对话记忆。

比如 `runtime_input.json` 里有：

```json
"selected_memory_ids": ["mem_conversation_conv_000"],
"use_global_memory": true,
"save_memory": "conversation"
```

意思是：运行前加载某些记忆；运行结束后把本轮对话保存成 conversation memory。

这个目录不要随便删。它是项目的“长期状态”。

**outputs 是啥**
`outputs/` 是每次运行生成的结果和日志。

比如完整 demo 会生成：

- `outputs/full_demo/final_answer.md`
- `outputs/full_demo/trace.json`
- `outputs/full_demo/messages.json`
- `outputs/full_demo/tool_messages.json`
- `outputs/full_demo/demo_report.md`
- `outputs/full_demo/llm_calls/...`

这个目录可以理解为“运行产物”。一般可以删了重新生成，不是核心输入。

**prompts 是啥**
`prompts/local_tool_agent.txt` 是系统提示词，也就是 Agent 的行为规则。

它不是用户问题，而是告诉模型：

- 需要时使用工具
- 不要编造文件内容
- 一次只调用一个工具
- 工具结果回来后再回答

`runtime_input.json` 里通过这个字段引用它：

```json
"system_prompt_path": "../prompts/local_tool_agent.txt"
```

一般这个可以固定不动。你改它会影响 Agent 行为风格和工具调用习惯。

**用户输入在哪**
用户输入在 `data/runtime_input.json` 的：

```json
"user_input": "帮我阅读 docs/agent_intro.txt，总结三条中文要点。"
```

当前项目没有交互式输入框，也没有 REPL。你要换问题，就是改 `runtime_input.json`，或者新建一个 `runtime_input_xxx.json`，然后运行时传进去。

例如：

```bash
python code/run_full_demo.py --input data/runtime_input.json --tools_config configs/tools.yaml --memory_config configs/memory.yaml --model_config configs/model.yaml --llm_mode prompt_json --outdir outputs/full_demo
```

如果你写了新任务：

```bash
python code/run_full_demo.py --input data/runtime_input_2.json --tools_config configs/tools.yaml --memory_config configs/memory.yaml --model_config configs/model.yaml --llm_mode prompt_json --outdir outputs/full_demo_calc
```

一句话总结：  
`configs` 是参数，`data` 是任务和工具素材，`memory` 是长期记忆，`prompts` 是系统规则，`outputs` 是运行结果，用户问题写在 `data/runtime_input*.json` 里。


models/
这是预留的本地模型权重目录。现在只有 .gitkeep，是为了让空目录能进 Git。当前你走的是远程 FastAPI，所以本地 models/ 没用。
如果以后要把模型权重放进项目里，本地 transformers 可以指向这里。但一般不建议把大模型权重塞进项目仓库。

skills/
是自定义工具能力目录，没错。现在已有：

calculator.py
file_reader.py
local_file_search.py
table_analyzer.py
format_converter.py
但不是“随便往里堆就自动可用”。新增 skill 至少要做两件事：

在 skills/ 里写函数。
在 configs/tools.yaml 里注册它的 module、function、参数 schema、required、returns，并加入 toolsets.basic_tools 或新 toolset。
否则 B3 不知道这个工具存在，模型也拿不到它的 schema。

code/
是核心代码目录。五个模块确实是主干：

b1_agent_runtime.py：Agent 主循环
b2_run_skill.py：单独跑 skill
b3_tool_layer.py：工具 schema 和工具执行
b4_local_agent_llm.py：模型调用和 AIMessage 解析
b5_memory.py：记忆加载和保存
common/
是公共工具轮子，路径、IO、schema、日志这些。正常业务开发基本不用动。只有你要改消息格式、SkillResult 格式、路径安全规则时才会碰。

run_full_demo.py
它现在是完整 demo 启动器，不是核心框架本身。以后做前端/后端服务化时，大概率不会直接用它作为线上入口。

但它不一定要“抛弃”。更准确说：

开发/验收/命令行测试：继续用 run_full_demo.py
真正接前端：应该直接调用 b1_agent_runtime.run_agent(...)，或者封装一个新的 API 服务入口
run_full_demo.py 可以留作 smoke test 和演示脚本
所以你的理解可以收敛成：

configs/  参数
data/     测试输入 + 工具读取的数据
memory/   长期记忆
models/   预留模型权重目录，现在基本不用
prompts/  系统提示词
skills/   可扩展工具函数
code/     核心框架
common/   公共轮子
outputs/  运行产物
run_full_demo.py  当前命令行 demo 启动器
后续业务化重点就是：加/改 skills，更新 tools.yaml，换业务数据路径，写一个真正给前端调用的 API 入口。
