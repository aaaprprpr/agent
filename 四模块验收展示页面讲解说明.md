# 四模块验收展示页面讲解说明

本文档用于配合当前前端验收展示页面进行现场讲解。当前页面已经完成 B1、B2、B3、B5 四个模块展示；B4 独立展示页暂未完成，讲解时需要如实说明：B4 的能力目前通过主对话链路、B1 执行轨道和 B3 工具调用闭环间接体现，后续再补独立页面。

这份说明不是 README，也不是代码文档，主要用于现场按页面讲清楚“这个模块做什么、页面在展示什么、它和其他模块如何协作”。

## 一、整体讲解顺序

建议现场先用主聊天页跑一个简单问题，再切换到模块展示页。

推荐流程：

1. 在主聊天页发起一个能触发工具调用的问题，例如读取本地文档、计算、生成文件或运行 Python 代码。
2. 等回答结束后，先切到 B1，讲 Agent 的整体运行轨道。
3. 再切到 B2，讲具体 Skill 的执行结果。
4. 再切到 B3，讲 tool_calls 如何被解析、校验、分发和包装。
5. 最后切到 B5，讲记忆系统的分层、压缩、召回和上下文注入思路。

讲解主线可以这样概括：

> 这个系统不是单纯把用户问题发给模型，而是由 B1 负责 Agent 运行流程，B4 负责模型决策，B3 负责工具协议解析和调度，B2 负责真正执行 Skill，B5 负责记忆和上下文。前端验收页把这些模块在一次对话中的协作过程拆开展示。

## 二、B1 页面讲解：Agent 运行与消息管理

B1 页面主要展示 Agent 的总控流程。它不是展示某个工具细节，而是展示一次对话如何从用户输入变成最终回答。

### 观察模式讲解

页面左侧是执行轨道，可以按顺序讲：

1. `输入门控`：用户输入进入系统后，B1 先接收 HumanMessage。
2. `读取 memory`：B1 会在运行前接收 B5 提供的 memory context。
3. `消息装配`：B1 把 system prompt、memory context、用户输入和后续消息组织成 messages。
4. `LLM 决策`：B1 调用 B4，得到 AIMessage。
5. `工具调用解析`：如果 AIMessage 中有 tool_calls，交给 B3 解析。
6. `Skill 执行`：B3 调用 B2 执行具体工具。
7. `状态收束`：工具结果回到 messages 后，B1 继续推进，直到最终回答。
8. `写入记忆`：本轮结束后，B5 记录对话和记忆信息。

中间区域是消息流，重点说明四类消息：

- `HumanMessage`：用户输入。
- `AIMessage`：模型输出，可能包含最终回答，也可能包含 tool_calls。
- `ToolMessage`：工具执行结果，作为下一轮模型输入。
- `MemoryContext`：B5 提供的历史上下文和记忆信息。

右侧 Workspace 快照用于说明：

- 当前会话 ID 和状态。
- 消息数量。
- 工具轮数。
- 当前轮最后输入和最后回答。
- 当前是否已经有可观察的 B3/B2 中间过程。

### 演示模式讲解

B1 演示模式当前用于构造 B1 的边界输入，包括：

- `conversation_id`
- `user_input`
- `selected_memory_ids`
- `toolset`
- `max_turns`

这里需要说明：当前页面的演示模式主要用于展示 B1 输入结构，运行按钮还没有接后端 demo API，不要说成已经能单模块执行。

### 可以强调的验收点

- B1 支持多轮消息组织。
- B1 能展示 LLM → Tool → LLM 的闭环。
- B1 负责 max_turns 和运行状态控制。
- B1 和 B2/B3/B5 是解耦的，B1 不直接执行 Skill，也不直接操作底层记忆库。

## 三、B2 页面讲解：Skill 工具函数模块

B2 页面主要展示“具体工具执行”。它回答的问题是：模型决定调用工具以后，真正执行的是哪些 Skill，输入输出是什么，是否成功。

### 观察模式讲解

页面顶部展示当前对话中的工具执行统计：

- 总执行次数。
- 成功次数。
- 失败次数。

左侧是执行时间线，每一项对应一次 Skill 调用。可以点击某个工具调用，右侧会展示详情。

右侧调用详情重点讲：

- `tool name`：被调用的 Skill 名称。
- `status`：执行成功、失败或等待。
- `latency`：耗时。
- `input`：B3 传给 B2 的参数。
- `output`：Skill 返回的 JSON 可序列化结果。
- `error`：异常或错误信息。
- `raw`：原始工具消息内容。

最右侧工具列表用于说明当前系统已经注册的 Skill，例如：

- calculator
- file_reader
- local_file_search
- table_analyzer
- file_writer 系列
- current_time
- web_search
- python_sandbox

### 演示模式讲解

B2 演示模式可以选择工具并填写 args JSON，用于展示“单个 Skill 的调用请求应该长什么样”。

这里也要如实说明：当前演示模式还没有接后端单模块执行 API，因此它主要用于展示请求结构和工具参数，不是现场执行入口。

### 可以强调的验收点

- 每个 Skill 都有明确输入参数和 JSON 化输出。
- B2 只负责函数能力，不负责模型决策。
- B2 的返回结果会被 B3 包装成 ToolMessage，再回到 B1。
- 工具执行错误不会直接摧毁主流程，而是作为结构化结果返回，方便 Agent 判断下一步。

## 四、B3 页面讲解：工具调用协议与调度模块

B3 页面主要展示 tool_calls 闭环。它介于 B4 和 B2 之间，负责把模型生成的工具调用转换为真实 Skill 执行，再把结果包装回 Agent 消息。

### 观察模式讲解

B3 页面按 Round 展示工具调用闭环。每个 Round 代表一次 assistant message 中出现 tool_calls，并经过 B3/B2 处理。

可以按照页面三段讲：

1. `LLM 侧`
   - B4 生成 AIMessage。
   - AIMessage 中包含 tool_calls。
   - B3 读取这些 tool_calls。

2. `B3 处理`
   - 解析：读取 AIMessage.tool_calls。
   - 标准化：提取工具名、参数、call id。
   - 校验：检查工具是否存在、参数是否满足 schema。
   - 分发：调用 B2 中对应 Skill。
   - 包装：把执行结果封装成 ToolMessage / trace，返回给 B1。

3. `Tool 侧`
   - 展示每个工具调用的名称、状态、输入、输出、错误和耗时。

这页重点不是展示 Skill 的内部实现，而是展示“模型工具调用协议如何落地”。

### 演示模式讲解

B3 演示模式目前是后续规划入口：未来可以粘贴 AIMessage/tool_calls，直接演示 B3 的解析、校验和包装流程。

当前需要如实说明：演示模式暂未接后端执行。

### 可以强调的验收点

- B3 不负责写具体工具逻辑，具体能力在 B2。
- B3 不负责模型调用，模型输出来自 B4。
- B3 是工具调用协议层，核心价值是 schema、校验、分发、包装、日志。
- 这个页面能展示至少一次 `LLM -> Tool -> LLM` 闭环中的工具调度过程。

## 五、B5 页面讲解：记忆文档存储与查找模块

B5 页面当前是独立展示设计，用于说明记忆系统的层级结构和召回逻辑。它还没有连接后端实时数据，因此现场讲解时必须实事求是：这是按照当前 B5 设计和测试样例制作的展示页，不是直接读取 SQLite 的实时页面。

### 页面整体讲法

B5 页面分为两个主要部分：

1. `历史记录与压缩`
2. `召回与上下文`

这两个部分对应 B5 的两条主线：

- 记忆如何沉淀：原始对话 → 轮级摘要 → 块级摘要。
- 记忆如何使用：当前输入 → 候选池 → 召回命中 → 源证据 → 拼给 B1 的上下文。

### 历史记录与压缩

左侧展示“全量数据库”，每个 turn 包含：

- 用户输入。
- AI 回答。
- 工具引用数量。

中间展示“轮对话压缩”，每一轮生成 turn_summary，并带有标签，例如：

- file:docx
- task_state
- preference
- requirement
- frontend

右侧展示“块级压缩”，多个相关 turn 会合并成一个 block，例如：

- 文件生成与附件展示。
- 模块验收界面整理。

这部分要强调：

> B5 不会只保存一段简单聊天摘要，而是保留原始消息，同时建立轮级摘要和块级摘要。摘要的作用是定位历史，不替代原始事实。

### 召回与上下文

右侧召回流程展示：

1. 当前输入形成 query text。
2. B5 从 blocks / turns / task memory 中构造候选池。
3. 根据关键词、任务相关度、长期价值、向量召回和 rerank 等信号排序。
4. 命中内容后回查 source messages / tool steps。
5. 组装 memory_messages，交给 B1。

召回内容列表展示不同类型的命中：

- task：当前任务状态。
- turn：相关轮级摘要。
- block：相关块级摘要。

上下文框展示 B5 拼给 B1 的内容，例如：

- 当前任务。
- 最近原始历史。
- 召回摘要。
- 策略说明。

这里可以强调：

> B5 的目标不是把全部历史都塞给模型，而是在保证最近上下文完整的基础上，对远距离历史做相关召回。精确事实仍然要回查 source message 或 tool step，避免摘要造成事实偏差。

### 演示模式讲解

B5 演示模式用于展示 `prepare_workspace_memory_context` 的输入形态：

- `conversation_id`
- `current_user_input`

当前运行按钮未接后端，因此只作为单模块输入结构说明。

### 可以强调的验收点

- B5 支持历史记忆保存。
- B5 支持轮级摘要和块级摘要。
- B5 支持任务记忆。
- B5 支持召回并将 memory context 提供给 B1。
- 当前页面展示的是结构和机制，后续可接入真实后端数据。

## 六、B4 当前如何说明

当前没有独立 B4 展示页，讲解时不要回避，可以直接说明：

> B4 独立展示页还没有完成，但 B4 的能力已经在主链路中体现：B1 的执行轨道中有 LLM 决策阶段，B3 页面中的 AIMessage/tool_calls 也来自 B4 的模型输出。后续 B4 页面会单独展示 model.yaml、模型模式、原始输出、AIMessage 解析和工具绑定方式。

如果老师追问 B4，可以从系统实现角度回答：

- B4 读取 `configs/model.yaml`。
- B4 接收 messages 和 tools_schema。
- B4 负责调用模型。
- B4 将模型原始输出解析成标准 AIMessage。
- 如果 AIMessage 包含 tool_calls，B1 会把它交给 B3。

不建议说“B4 页面已经完成”，也不建议把 B3 的页面说成 B4 页面。

## 七、现场讲解建议话术

可以用下面这段作为整体开场：

> 我们的 Agent 系统按 B1 到 B5 拆分。B1 负责运行编排和消息管理，B2 负责具体 Skill，B3 负责工具调用协议和调度，B4 负责模型决策，B5 负责记忆和上下文。当前前端验收页已经做了 B1、B2、B3、B5 四个模块展示，B4 独立页面还没完成，所以 B4 暂时通过 B1 的 LLM 决策阶段和 B3 的 tool_calls 闭环来说明。

B1 过渡到 B2：

> B1 页面看到的是完整运行流程，但它并不执行工具。工具真正执行的细节在 B2 页面。

B2 过渡到 B3：

> B2 只说明函数执行结果。模型为什么选择这个工具、tool_calls 如何变成 Skill 调用，需要看 B3。

B3 过渡到 B5：

> B3 解决工具调用闭环，B5 解决历史上下文。Agent 不是每次只看当前输入，而是通过记忆模块把任务状态、历史偏好和相关旧对话组织进上下文。

B5 收尾：

> 所以整个系统形成了一个闭环：B1 编排，B4 决策，B3 调度工具，B2 执行能力，B5 提供长期记忆。前端页面的作用是把这些模块边界和数据流拆开展示，方便验收时逐项说明。

## 八、需要避免的说法

以下说法不建议现场使用：

- 不要说 B4 独立展示页已经完成。
- 不要说 B2/B3/B5 演示模式已经全部接入后端执行。
- 不要说 B5 页面当前读取的是实时数据库内容。
- 不要说摘要替代了原始消息。
- 不要说前端页面本身就是模块实现，前端只是验收展示层。

更准确的说法是：

- B1/B2/B3 观察模式能基于当前会话展示真实运行状态。
- B5 当前展示的是记忆系统结构和运作机制，后续可以接真实后端数据。
- 演示模式主要用于展示模块输入输出边界。
- 模块真正实现仍在后端代码、配置文件和运行产物中。
