## 一、验收总体策略

验收安排中每组预计约 15 分钟，建议不要把时间全部花在运行过程上。最稳的结构是：

1. 先由组长展示整体系统效果。
2. 再由五个模块负责人分别讲清楚自己模块的职责、输入输出、与其他模块的接口。
3. 每个模块都准备一份可打开的证据文件：代码入口、配置、样例输入、运行产物。
4. 老师提问时优先回答“模块边界”和“数据如何流动”，不要只讲 UI。

建议现场展示顺序：

1. 前端整体演示：输入问题，展示流式回答、工具过程、生成文件下载、历史会话。
2. 打开一次运行产物：`outputs/backend_runs/<conversation_id>/<run_id>/trace.json`。
3. 依次讲 B1、B2、B3、B4、B5。
4. 最后总结：系统不是五个孤立脚本，而是 B1 编排、B3/B2 执行工具、B4 决策、B5 提供上下文和长期记忆。

## 三、B1 验收方法：Agent 运行与消息管理

### 1. 现场要讲清楚

B1 是 Agent 的主编排模块，职责是接收用户问题，读取系统提示词和记忆上下文，获取工具 schema，调用模型生成 AIMessage，判断是否有 tool calls，再交给 B3 执行工具，并把 ToolMessage 追加回 messages，直到得到最终回答。

重点强调：

- B1 不直接执行 Skill。
- B1 不直接调用底层模型。
- B1 不直接写长期记忆。
- B1 只负责流程、状态、消息序列和运行产物。

当前项目中 B1 已经从基础 ReAct 循环扩展为 workspace 分阶段循环：

- planning
- tool_calling
- observation
- answering

这可以解释为对基础 Agent Loop 的增强，但不要声称完全实现了某篇论文的完整 Plan-and-Execute 框架。

### 4. 老师可能问

问：SystemMessage、HumanMessage、AIMessage、ToolMessage 分别是什么？

答：

- SystemMessage：约束 Agent 行为、工具使用规则和输出格式。
- HumanMessage：用户当前输入和任务目标。
- AIMessage：模型生成结果，可能是最终回答，也可能包含 tool_calls。
- ToolMessage：工具执行后的结构化结果，由 B3 返回给 B1，供下一次模型调用参考。

问：为什么 B1 不直接调 Skill？

答：为了保持边界清晰。B1 是编排层，B3 是工具层，B2 是具体工具函数。这样工具校验、日志、缓存、重试都集中在 B3，B1 不会被具体工具污染。

### 4. 老师可能问

问：Skill 和 Tool Schema 区别是什么？

答：Skill 是实际执行的 Python 函数，Tool Schema 是给模型看的工具说明，包括工具名、描述、参数和返回值。模型只看 schema 生成 tool_calls，真正执行由 B3 调用 Skill。

问：文件写入安全吗？

答：文件生成只写到当前 run 的 `generated_files/` 下，拒绝绝对路径、`..`、Windows 保留名和非法后缀，不覆盖同名文件。

## 五、B3 验收方法：说明生成与工具调用

### 1. 现场要讲清楚

B3 是 Skill 和 LLM 之间的桥接层，负责：

1. 从 `tools.yaml` 读取 toolset。
2. 生成 OpenAI-style tools schema。
3. 校验模型生成的 tool_calls。
4. 动态加载 B2 Skill 并执行。
5. 把 SkillResult 封装为 ToolMessage。
6. 写入工具调用日志、统计和缓存。

B3 不决定是否调用工具，也不生成自然语言回答。

### 4. 老师可能问

问：B3 如何防止模型乱调用不存在的工具？

答：B3 会检查 tool name 是否属于当前 toolset，并校验 required 参数和基础类型。不存在或参数错误会返回 error ToolMessage，而不是直接崩溃。

问：为什么 B3 要生成下载链接？

答：文件生成结果是工具真实产物，B2 返回相对产物路径，B3 根据当前 run 的 `output_dir` 生成受限后端下载 URL，前端再展示下载卡片。模型不能凭空生成下载链接。

## 六、B4 验收方法：Agent LLM 决策

### 1. 现场要讲清楚

B4 是模型桥接和输出解析模块。它接收 messages 和 tools_schema，调用模型，得到原始输出，再解析成标准 AIMessage。

B4 的职责边界：

- 不执行工具。
- 不决定工具执行结果。
- 不写 memory。
- 不做业务级硬编码判断。
- 只负责模型通信、prompt 注入、格式约束、AIMessage 解析和日志。

当前支持模型来源：

- `fastapi`
- `qwen_api`

### 4. 老师可能问

问：什么是 AIMessage？

答：AIMessage 是 B4 输出给 B1 的标准结构。它要么包含最终 `content`，要么包含 `tool_calls`。B1 只看这个结构决定下一步，不直接依赖模型原始文本。

问：模型输出 JSON 格式错怎么办？

答：B4 会记录 raw output 和解析错误，尽量做结构层恢复；如果无法恢复，会返回可诊断错误，B1 再根据阶段决定是否失败或兜底回答。

### 4. 老师可能问

问：为什么 `memory_index.json` 不是主线？

答：课程基础版用 `memory_index.json` 管理 markdown 文档，适合最小演示。但我们前端多轮对话需要更细粒度的历史、工具步骤、任务状态和召回，因此主线升级为 SQLite。`memory_index.json` 仍保留兼容样例和旧命令演示，不影响新系统。

问：摘要会不会造成幻觉？

答：B5 的设计要求摘要只用于定位。真正涉及精确事实时，会加载 source message 和 source tool step。当前输入和原始来源优先于旧摘要。

问：向量召回失败怎么办？

答：B5 的向量和 rerank 都是可降级链路。embedding 服务不可用、rerank 输出非法 JSON 时，B5 会记录状态并回退到确定性排序，不中断 B1 回答。

## 十二、最短讲解稿

如果现场时间很紧，可以按下面说：

```text
我们的系统按 B1-B5 解耦实现。B1 是主编排，维护 messages 和 Agent Loop；B2 是具体 Skill；B3 从 tools.yaml 生成 schema 并执行工具；B4 负责模型调用和 AIMessage 解析；B5 负责记忆。

相比课程基础版，我们保留了每个模块的独立命令行入口和 JSON 产物，同时做了前端、后端、流式输出、终止回答、文件上传、文件生成下载和 SQLite 分层记忆。

工具调用链路是：B1 把 messages 和 tools_schema 交给 B4，B4 输出 tool_calls，B1 交给 B3，B3 调 B2 Skill 并返回 ToolMessage，B1 再把 ToolMessage 加回 messages 调 B4 得到最终回答。

B5 方面，基础的 memory_index.json 兼容保留；主线升级为 SQLite 分层记忆，保存原始消息、工具步骤、轮级摘要、块级摘要和任务记忆，召回时通过关键词/字段评分、向量 RAG 和 LLM rerank 选择相关历史，并加载 source message/tool step 保证精确信息有来源。
```
