本地 Agent 智能体实训系统内部教程

文档用途

本文面向刚加入项目、尚未参与设计和开发的团队成员。目标不是介绍项目成果，而是说明系统为什么这样设计、五个模块如何协作、校方要求完成到什么程度、哪些能力采用了等价实现、哪些内容仍有缺口，以及修改代码时必须守住哪些边界。

本文依据当前仓库、B 方向 PPT、校方说明文档、现行 README、成员开发记录、配置文件和模块源码整理。项目处于收尾阶段，判断实现情况时应以当前代码和实际运行产物为准，不以早期文档或旧样例为准。


一、先理解项目解决了什么问题

这个项目实现的是一个能够连续对话、调用工具、保存记忆并交付结果的 Agent 系统。它不是普通聊天页面，也不是把用户问题直接转发给模型。一次完整任务至少包含四类工作：

1. 接收目标，结合系统提示词和历史信息理解当前任务。
2. 判断是否需要外部能力，例如读文件、计算、查询时间或生成文件。
3. 执行工具，把真实结果返回给模型继续判断。
4. 输出最终回答，并保存消息、工具过程、运行轨迹和可供后续对话使用的记忆。

项目按照校方要求拆成 B1-B5 五个模块。最重要的不是记住文件名，而是先记住职责：

B1：总控。维护消息和任务状态，决定下一步调用 B4 还是把工具请求交给 B3。
B2：能力。实现计算、文件、搜索、表格、时间、代码执行等具体工具。
B3：工具协议。把 B2 描述成模型能理解的工具说明，并校验、执行模型发出的工具请求。
B4：模型接口。把消息和工具说明交给模型，再把模型原始输出整理成标准 AIMessage。
B5：记忆。保存会话事实，生成摘要和任务记忆，为后续对话召回相关上下文。

浏览器、FastAPI 后端和模型代理是五模块之外的工程层。它们负责把课程模块变成可连续使用和现场展示的完整产品，但不能取代 B1-B5 的职责。


二、完整链路

最常见的工具任务流程如下：

用户在前端发送问题
  -> 后端整理会话、上传文件和系统提示词
  -> B1 接收本轮输入
  -> B5 提供近期原文和相关长期记忆
  -> B3 提供当前工具说明
  -> B1 调用 B4 进行规划和决策
  -> B4 返回包含工具请求的 AIMessage
  -> B1 将工具请求交给 B3
  -> B3 校验后调用 B2
  -> B2 返回结构化结果
  -> B3 包装成 ToolMessage
  -> B1 把 ToolMessage 加入消息和工作区
  -> B4 根据工具证据生成最终回答
  -> 后端流式返回前端
  -> B5 在后台保存本轮事实、摘要和任务状态

不需要工具时，B4 会直接给出最终内容，B1 不进入工具执行环节。需要多个工具时，同一个 AIMessage 可以包含多个工具请求，B3 会逐项处理；任务也可以经过多轮“模型判断—工具执行—模型观察”，直到完成或达到最大轮数。

项目当前的主运行方式不是最初版本的单层循环，而是 B1 workspace 分阶段流程：

planning：识别目标、要求、成功条件和是否需要工具。
tool_calling：根据计划生成一个或多个工具请求。
observation：检查工具是否成功、证据是否可信、是否还缺信息。
answering：只根据已接受的证据和当前任务状态组织最终回答。

这种设计把“想做什么、实际做了什么、结果是否可用、最后怎么回答”分开记录，比模型一次性决定全部步骤更容易排查，也减少工具失败后仍然声称成功的问题。


三、消息、工具和记忆的基本概念

1. 四种消息

SystemMessage：系统规则，规定身份、行为边界、工具使用原则和输出要求。
HumanMessage：用户输入，包括当前问题和历史用户消息。
AIMessage：模型输出。它可以是最终回答，也可以包含工具请求。
ToolMessage：工具执行结果，带有对应调用编号、工具名、成功或失败状态。

消息顺序不能随意调整。工具结果必须出现在提出该工具请求的 AIMessage 之后，否则模型无法把请求和结果正确对应。

2. Skill、Tool Schema 和 Tool Call

Skill 是可执行函数，例如计算器或文件读取器。
Tool Schema 是给模型看的工具说明，包括名称、用途、参数和返回字段。
Tool Call 是模型根据说明生成的调用请求，包含调用编号、工具名和参数。

B2 实现 Skill，B3 生成 Tool Schema 并执行 Tool Call。把这两层分开后，工具函数可以独立使用，模型也不能绕过 B3 的参数检查直接执行任意函数。

3. SkillResult

所有工具最终都整理成统一结果，主要包含工具名、状态、输入、业务输出、错误、耗时、来源和文件产物。正常错误也应成为结构化结果，例如除零、文件不存在、参数缺失或执行超时，而不是让整个 Agent 服务崩溃。

4. 记忆不是聊天记录的简单拼接

当前系统保留最近四轮原始对话。更早内容先通过轮级摘要、记忆块和任务记忆定位，再按需要加载对应原始消息和工具步骤。摘要只用于找到证据，不作为精确事实的唯一来源。这一原则能减少摘要失真对最终回答的影响。


四、目录如何阅读

建议先看以下目录，不要从前端大文件开始反推全部逻辑。

README.md
  对外的项目总说明，适合了解运行环境、命令、输入和输出。

code/
  B1-B5 的课程核心代码。五个入口文件仍然保留独立命令行能力。

code/b1_agent_runtime_parts/
  B1 拆分后的工作区、分阶段提示、检查点、旧循环和输入校验。

code/b5_memory_parts/
  B5 拆分后的旧版文档记忆、SQLite 分层记忆、反思、召回、向量检索和重排。

code/common/
  公共消息结构、配置读取、路径处理、日志和 SQLite 会话存储。这里是跨模块的数据契约，不属于某个工具的业务实现。

skills/
  B2 的具体工具。新增工具时通常先在这里实现，再到工具配置中注册。

configs/
  三个关键边界：模型、工具、记忆。修改配置可能同时影响多个模块，必须理解字段用途后再改。

prompts/
  Agent 系统提示、B1 各阶段提示和 B5 记忆提示。提示词属于运行逻辑的一部分，不是普通文案。

backend/
  FastAPI 接口、流式运行、会话管理、上传、模块展示接口和文件下载。

frontend/src/
  主对话界面、B1-B5 观察/演示页、流式事件处理和下载展示。模块页主要负责展示真实状态，不应复制一套课程模块逻辑。

llm_backend/
  本地模型服务和 Qwen API 代理。B4 只依赖统一的生成接口，不直接关心代理内部实现。

data/
  文档、表格、工具输入、消息输入和 B1 运行样例。

memory/
  旧版文档记忆样例和当前 SQLite 会话记忆库。两套实现的用途不同，不能混为一谈。

outputs/
  各次运行证据，包括消息、轨迹、模型原始输出、工具日志、生成文件和模块演示结果。

B方向/
  校方 PPT、教程和模板，只用于对照要求，不参与程序运行，也不应按项目代码随意修改。


五、B1：Agent 运行与消息管理

1. 模块职责

B1 是唯一的主流程编排者。它接收一轮用户问题，准备系统提示、历史和记忆，获取工具说明，调用 B4，判断是否需要工具，调用 B3，维护消息顺序，并生成本轮输出文件。

B1 不实现计算器，不直接读取文件，不解析模型服务协议，也不自行生成长期记忆。出现这类逻辑时，应先检查是否放错模块。

主要入口是 code/b1_agent_runtime.py。实际分阶段循环位于 code/b1_agent_runtime_parts/b1_workspace_loop.py，工作区结构位于 b1_workspace.py，各阶段模型输入位于 b1_prompting.py。

2. 校方基础要求完成情况

接收一个用户问题：已实现，CLI 和浏览器均可进入 B1。
调用 B5 获取记忆：已实现，包括旧版记忆读取和当前 SQLite 分层召回。
构造初始消息：已实现，包含系统提示、近期历史和当前用户输入。
获取工具说明并交给 B4：已实现。
接收并判断 AIMessage：已实现。
完成至少一次“模型—工具—模型”闭环：已实现，且支持多轮工具循环。
限制最大工具轮数：已实现，超过限制会以可解释状态结束。
保存消息、轨迹和最终回答：已实现。
说明四类消息作用：代码结构和前端 B1 页面均可观察。

需要注意：旧版 memory 文档的读取、截断和 selected_memory.json 已实现；当前 prompt_json workspace 主链路主要使用 SQLite 分层记忆。旧版文档正文目前不会直接作为活动 workspace 的完整内容注入，workspace 中保留的是旧版记忆概览，主要上下文来自近期原文、任务记忆、召回轮次和来源证据。验收时不能把“旧版文档可读取”与“旧版正文在新工作区直接注入”说成同一件事。

3. 校方进阶要求完成情况

多轮用户输入：已实现。浏览器会话保存历史，B5 负责长期上下文。
多次工具循环：已实现。每轮可以有一个或多个工具请求。
断点续跑：代码已实现停止、检查点和恢复入口，能保存阶段、消息、工具轮次和工作区后继续；严格的可控中断实验和完整恢复报告尚未收口，属于“功能存在、验证仍需补齐”。
批量任务运行：未实现独立批量 runner。现有 run_full_demo 仍是单任务汇总入口。
历史消息压缩：采用 B5 分层记忆等价实现。最近历史保留原文，较早历史使用轮摘要和记忆块定位，并回查来源；不是在 B1 内把全部历史替换成单段摘要。
系统提示模板切换或添加：部分实现。每个会话可读取默认提示并保存自定义提示，支持会话级替换；尚未提供完整的多模板库和自动切换策略。

4. B1 的主要优化

从简单 ReAct 循环升级为 planning、tool_calling、observation、answering 四阶段工作区。
明确保存成功证据、失败证据、已知事实、缺失信息和必需产物，减少模型忽略工具错误。
加入流式事件，前端可以看到规划、工具和最终回答进度。
加入取消与恢复，取消的回答不会写入完成态记忆。
工具已经成功但观察阶段解析失败时，优先基于成功工具证据生成正常回答，避免向用户暴露内部解析错误。
运行轨迹保留每次模型调用和工具轮次，便于从产物定位问题。

5. 新成员阅读重点

先看 b1_agent_runtime.py 的入口选择，再看 workspace 的数据结构，最后顺着 workspace_loop 的四阶段读。不要一开始逐行阅读近两千行循环代码。判断 B1 问题时，先分清是输入准备、模型输出、工具结果、观察判断还是最终回答阶段。


六、B2：Skill 工具函数

1. 模块职责

B2 只负责把明确参数转成确定性结果。它不知道当前对话目标，也不决定是否调用工具。每个 Skill 应能独立接收参数、返回可 JSON 序列化结果，并把异常交给统一结果包装。

当前 basic_tools 共注册 15 个工具，可分为五组：

基础计算与实时信息：calculator、current_time。
文件探路和读取：directory_list、file_stat、file_reader、local_file_search。
数据处理：table_analyzer。
文件生成：纯文本、Markdown、代码、JSON、Word、CSV/TSV 六类 writer。
外部和执行能力：web_search、python_sandbox。

file_reader 除 txt、md 等文本外，还支持 docx、pptx 文本提取；table_analyzer 支持 CSV、TSV 和 XLSX；目录和文件工具都受允许工作区限制。

2. 校方基础要求完成情况

至少五个 Skill：已超过要求。
明确参数、描述和返回值：由工具配置统一维护。
结果可 JSON 序列化：已实现统一 SkillResult。
独立命令行入口：已保留。
正常和异常样例：data/tool_inputs 中已提供多类输入。
可被 B3 加载并记录日志：已实现。

3. format_converter 的替代关系

校方推荐 format_converter，把文本转成 Markdown 或 JSON 文件。项目没有保留同名配置工具，而是拆成多种专用 writer。这样做有三个实际好处：

一是每个工具只接受与目标格式匹配的数据，例如 JSON writer 接收对象，避免模型手写非法 JSON。
二是每种文件后缀和返回字段都能单独约束。
三是写文件工具统一限制在本次运行的 generated_files 目录，并且不覆盖同名文件。

这属于功能替代，不是同名实现。校方 PPT 使用的是“推荐”工具，但现场说明时应明确替代关系，不能直接称为已实现 format_converter。同样，历史 runtime_input_5.json 仍使用旧工具名，不适合作为当前能力证明。

4. 校方进阶要求完成情况

增强本地检索：已实现关键词拆分、路径和正文匹配、片段截取、评分和 top-k 排序；这是轻量本地文本检索，不是向量文件搜索。
代码执行 Skill：已实现 python_sandbox。它使用独立运行目录、隔离参数、超时、输出限制和进程终止，但不是容器级安全沙箱，不能执行不可信攻击代码。
复合 Skill：未实现一个“读取后转换”的单独复合函数；系统通过 B1 多步调用 file_reader 和对应 writer 达到相同任务效果。这是系统层等价编排，不是 B2 单工具层的原要求实现。
完善错误分类：部分实现。工具统一返回成功/失败、异常类型和消息，Python 工具还有完成、非零退出、超时等诊断；尚未建立统一数字错误码体系。
高耗时和高风险限制：已实现一部分。文件路径受限、写文件不覆盖、联网搜索有明确错误返回、Python 有时限和输出上限；强隔离仍需容器或低权限运行环境。

5. B2 的主要优化

增加目录探路和文件状态工具，避免模型在不知道文件名时盲猜路径。
Office 文档读取使用标准压缩包和 XML 解析，不依赖桌面 Office。
文件生成结果附带产物信息，经过 B3 和后端转成前端下载入口。
写文件工具不修改项目源码，只写本轮输出目录。
联网搜索直接使用 DDGS/DuckDuckGo，降低额外付费接口依赖；网络不可用时返回错误，不伪造结果。


七、B3：工具说明生成与调用执行

1. 模块职责

B3 位于模型决策和工具函数之间。它负责两件事：

第一，从工具配置中选择当前 toolset，生成 OpenAI 风格工具说明。
第二，接收一个或多个工具请求，校验工具名、必填参数和基础类型，调用 B2，再生成 ToolMessage。

B3 不判断用户想做什么，也不把工具结果改写成最终自然语言回答。

2. 校方基础要求完成情况

读取 tools.yaml 和 toolset：已实现。
生成包含名称、描述、输入和返回信息的 tools schema：已实现，并增加返回结构和 SkillResult 说明。
接收 tool_calls：已实现。
校验工具存在、参数完整和基础类型：已实现。
调用 B2 并生成 ToolMessage：已实现。
保存 schema、消息和工具日志：已实现。

3. 校方进阶要求完成情况

从 Python 函数自动生成 schema：部分实现。当前仍以 YAML 为主，B3 会读取函数签名，自动补充 YAML 缺失参数和必填项，并在报告中记录来源；尚未做到完全脱离 YAML 自动生成名称、描述和完整参数语义。
可恢复错误有限重试：已实现。只重试配置中的可恢复异常，带副作用的写文件工具强制只执行一次。
工具结果缓存：已实现运行输出目录内的缓存。缓存键包含工具名、参数和执行上下文，只对配置允许的只读工具生效，不是跨所有会话的全局缓存。
工具统计：已实现调用次数、成功数、失败数、缓存命中、平均耗时和失败率；当前统计针对本次执行记录，不等同于大规模评测报告。
比较不同 schema 描述效果：未完成，缺少固定批量样例和准确率对照报告。

4. B3 的主要优化

工具结果统一附带摘要、来源和产物，使模型不必读取全部冗长输出才能判断结果。
未知工具和缺少参数会生成 error ToolMessage，主流程可以继续收束回答。
文件产物通过严格的相对路径检查生成下载地址，后端只开放 generated_files 下的内容。
B2、B3 展示页调用真实模块接口，不使用前端伪造结果；演示输入可以编辑。


八、B4：Agent LLM 决策

1. 模块职责

B4 只负责模型通信、工具说明注入和 AIMessage 解析。它不执行工具、不维护 Agent 循环、不保存长期记忆。B1 给它消息和工具说明，它返回标准 AIMessage 或结构化 JSON 结果。

项目支持三种模型来源：

local：本机或服务器直接加载 transformers 模型。
fastapi：连接统一的远程模型服务。
qwen_api：通过本地代理调用 Qwen API。

当前默认配置是 qwen_api，模型为 qwen-plus。校方指定的 Qwen3.5-4B 本地路径和加载能力仍保留，但仓库不包含模型权重。

这里必须区分“工程接口替代”和“模型相同”。qwen-plus API 让 B4 在本地算力不足时仍能完成工具决策，是当前运行替代方案；它不是 Qwen3.5-4B 本地模型本身。若验收严格要求指定模型，应切换到 local 配置并准备对应权重，不能把默认 API 路径宣称为同模型部署。

2. 校方基础要求完成情况

读取 model.yaml：已实现。
接收工具说明并完成绑定：已实现，当前主要采用 prompt_json 注入。
接收 messages 并调用模型：已实现。
解析 content 或 tool_calls：已实现。
保存模型原始输出和标准 AIMessage：已实现。

3. 校方进阶要求完成情况

单轮多个工具请求和接收多个 ToolMessage：已实现，B4 协议、B3 执行和 B1 循环都支持。
Plan-and-Execute：在项目层采用 B1 workspace 分阶段实现。规划属于 B1 编排，B4 提供各阶段模型生成；不是 B4 内部另建一套流程。
按任务切换不同本地模型：部分实现。配置支持 local、fastapi、qwen_api 和不同模型路径，但当前没有根据任务内容自动路由模型。
模型内置工具参数与 prompt 注入对比：未完成。当前主线是 prompt_json。
不同模型工具成功率和 token 对比：未完成，缺少批量评测和统一 token 统计。

4. 协议容错与展示结果

小模型或远程模型可能输出纯文本、Markdown 标记、损坏 JSON，或者把工具参数写成 parameters/arguments。B4 当前会处理可明确恢复的格式偏差：

把 parameters 或 arguments 归一化为标准 args。
把明显的纯文本回答整理成 content。
允许简短说明和工具请求同时存在，并规范控制状态。
远程流式输出未产生中间 content 时，在最终解析成功后补发完整内容。
损坏 JSON、空消息和无法确认语义的内容仍返回错误，不静默伪造成功。

本地最新 B4 协议运行产物位于 outputs/backend_runs/b4_demo/20260714_110530_516977，记录为 10 项全部通过，使用 qwen_api/qwen-plus。这个结果证明当前配置下该次运行通过，不代表其他模型、网络状态或配置一定得到相同结果。

5. B4 的主要优化

统一 local、fastapi 和 qwen_api 的上层接口，B1 不需要跟随模型来源变化。
保留 prompt、原始输出、解析结果和错误，便于判断问题出在模型还是解析器。
为 Qwen API 的 prompt_json 请求启用 JSON 对象响应格式，减少协议漂移。
B4 观察页区分 Agent 主链路、记忆辅助调用和独立演示调用，避免把 B5 反思误认为用户回答。
B4 演示页同时包含真实模型用例和纯解析器用例，能分别检查模型服从性与解析稳定性。


九、B5：记忆存储与检索

1. 两套记忆实现

旧版课程兼容实现：memory_index.json 加 Markdown 文档。它支持按 memory id 读取、全局记忆选择、长度截断、保存 messages/trace/final answer、更新索引和写日志。它用于对应校方基础要求和独立 CLI 演示。

当前主要实现：conversation_store.sqlite3。它保存会话、原始消息、工具步骤、轮级摘要、记忆块、任务记忆、向量缓存和召回日志，服务于浏览器多轮对话。

两套实现并存不是重复开发。旧版保证课程要求可直接对照，SQLite 解决长对话、检索、更新和证据追踪问题。

2. 当前记忆层次

近期原文：最近四轮，直接进入当前工作区，不做摘要。
轮级摘要：每轮结束后提取主题、事实、决定、纠正、偏好和任务相关性。
记忆块：至少三轮后，根据主题切换、任务切换、阶段完成、任务完成、轮数或长度边界聚合；单块最多八轮。
任务记忆：区分前台任务、暂停任务、完成任务，记录目标、进度、决定和待办。
来源证据：摘要保留原始消息和工具步骤编号，需要精确事实时回查来源。
向量召回：调用 embeddings 接口，向量缓存在 SQLite 中，并与原有字段评分组合。
LLM 重排：只允许从候选编号中选择和排序，失败时回退原排序。

3. 校方基础要求完成情况

读取 memory 配置：已实现。
读取全局和指定 id 文档：旧版实现已完成。
限制最大长度并生成 selected_memory.json：已实现。
更新 memory_index 元信息：已实现。
保存当前消息、轨迹和回答为记忆文档：已实现。
区分全局和对话记忆：旧版实现已完成；SQLite 主线以会话为主要隔离边界。

4. 校方进阶要求完成情况

关键词检索排序和 top-k：在 SQLite 主线采用字段重合、工具重合、任务相关度、长期价值、时间新近度等评分实现，并结合模型生成的关键词和标签；旧版 Markdown 仍主要按指定 id 读取，不提供完整关键词排序。
长度管理和摘要：已实现轮级摘要、块级聚合和 2000 字符上下文预算，不直接把全部历史塞给模型。
指定记忆更新与冲突管理：部分实现。任务记忆支持更新、切换、暂停、恢复和完成，但尚未对任意指定 memory 文档提供明确的重复、补充、冲突比较和人工确认流程。
向量检索：已实现，依赖可用 embeddings 服务；不可用时回退非向量排序，不中断主回答。
错误记忆影响分析：尚未形成系统实验报告。已有来源证据、召回日志和重排机制，但不能替代正确/错误记忆对照实验。

5. B5 的主要优化

摘要只作定位，原始消息和工具步骤作为事实来源。
记忆反思在回答结束后由后台线程处理，前端不必等待摘要生成。
取消的回答不进入完成态记忆，避免半截内容污染后续对话。
闲聊和低价值噪声默认不更新任务记忆。
中文对话尽量生成中文摘要，模型反思失败时使用中性定位说明，不凭规则臆造事实。
向量服务或重排失败会降级，而不是让 Agent 主流程失败。
B5 页面把真实近期原文、轮摘要、记忆块、任务、召回记录和来源证据分开展示。


十、后端和前端如何承接五个模块

1. FastAPI 后端

后端不是新的 Agent 决策层。它负责把浏览器请求转换成 B1 输入，并提供以下工程能力：

会话创建、读取和删除。
系统提示词读取与会话级保存。
文件上传和上传目录清理。
普通与流式 Agent 运行。
回答停止与检查点恢复。
B1 工作区快照。
B2 独立工具展示。
B3 工具说明和工具请求展示。
B4 调用观察和协议用例。
B5 记忆快照和召回预览。
工具步骤读取与生成文件下载。

后端主运行服务在 backend/run_service.py。它负责保存前端消息、启动 B1、转发流式事件、记录工具步骤，并在回答完成后安排 B5 记忆处理。

2. React 前端

第一屏是实际对话工作区。用户可以创建和切换会话、发送文本、上传文件、停止或恢复回答、修改当前会话提示词、查看工具过程和下载生成文件。

B1-B5 页面都有观察和演示两种用途：

观察页：读取当前会话或最近运行的真实数据，解释主链路正在发生什么。
演示页：使用独立样例调用对应模块接口，便于展示正常、异常和边界情况。

B2、B3 演示会真实执行工具。B4 模型类用例会真实调用当前模型，解析器类只回放协议。B5 召回演示会真实调用召回接口；轮摘要和记忆块来自对话结束后的后台流程，不是按钮即时伪造。

前端曾按模块拆分 App、侧边栏、工作区、样例和通用展示函数，目的是降低单文件体积。B1ModuleView、B2ModuleView、B3ModuleView 和 B5ModuleView 仍然较大，后续整理只能继续拆展示组件，不能改变接口和数据含义。


十一、配置是模块边界，不是普通常量

1. configs/tools.yaml

决定哪些工具属于当前工具集、工具位于哪个模块、参数和返回值如何描述、哪些工具允许缓存、哪些错误可重试，以及可访问哪些工作区根目录。

新增工具时至少要同步：Skill 函数、工具配置、工具集注册、输入样例和前端演示样例。只写 Python 文件而不注册，B3 和模型都看不到该工具。

2. configs/model.yaml

决定 B4 使用 local、fastapi 还是 qwen_api，模型位置、服务地址、生成参数、上下文长度和产物保存策略。切换模型来源不应要求修改 B1、B2、B3 或 B5 接口。

3. configs/memory.yaml

决定旧版记忆目录、SQLite 路径、上下文字符预算、向量召回和 LLM 重排。向量和重排可以独立降级，不能把外部服务故障变成主对话硬故障。

4. prompts/

agent_system_prompts.json 规定普通对话行为。
b1_stage_prompts.json 规定 B1 各阶段输入和输出协议。
b5_memory_prompts.json 规定记忆反思、任务判断和重排规则。
conversation_prompts.json 保存会话级自定义提示，属于运行数据。

修改提示词后，必须同时检查 JSON 格式、字段名称、对应解析逻辑和模型输出协议。提示词中的一个错误示例可能让模型在所有轮次重复调用某个工具。


十二、从校方要求看当前完成度

以下结论适合内部判断，不应为了汇报把“部分实现”改成“完全实现”。

B1 基础要求：主体完成。消息管理、工具闭环和 B5 分层上下文均已接入；旧版 selected/global memory 文档正文在当前 workspace 中主要由 SQLite 分层上下文替代，兼容路径与 PPT 原始方案存在差异。
B1 进阶：多轮、多工具循环、分层历史和会话提示已实现或等价实现；断点恢复需补完整验证；批量任务未实现。

B2 基础要求：完成，工具数量和类型超过要求。
B2 进阶：本地检索、代码执行和风险限制已实现；复合 Skill 采用系统编排替代；统一错误码未完成。

B3 基础要求：完成。
B3 进阶：有限重试、运行内缓存和统计已实现；自动 schema 只完成函数签名补充；schema 描述效果实验未完成。

B4 基础要求：完成。
B4 进阶：多工具、流式和项目级 Plan-and-Execute 已实现；模型自动路由、内置工具参数对照和跨模型/token 统计未完成。

B5 基础要求：旧版兼容实现完成，当前主线已扩展为 SQLite 分层记忆。
B5 进阶：摘要、分层检索、向量和重排已实现；旧版文档关键词排序不是主线；显式冲突合并和错误记忆影响实验未完成。


十三、项目中的关键替代和创新

1. format_converter 被专用文件生成工具组替代。职责更细，参数更可靠，但不是同名工具。
2. 指定本地模型仍可配置，默认运行改为 qwen-plus API。解决本地算力问题，但不是同模型等价，验收表述必须区分。
3. 简单 Agent Loop 升级为分阶段 workspace。工具使用前有规划，工具使用后有观察和证据筛选。
4. B1 内单段历史摘要被 B5 分层记忆替代。保留近期原文，摘要负责定位，精确事实回查来源。
5. 单一 Markdown 记忆扩展为 SQLite 事实表、摘要、记忆块、任务状态、向量和召回日志，同时保留旧接口兼容。
6. 文件生成从“模型在回答里给文本”升级为真实落盘、受限下载和前端产物卡片。
7. 五模块从 CLI 演示扩展到浏览器真实观察页，每个页面尽量读取模块产物，而不是展示静态说明。
8. B4 增加协议容错，但只修复能确定的格式问题，不根据用户关键词凭空制造工具请求。


十四、常见修改如何落位

新增一个工具：放在 skills，注册到 tools.yaml，通过 B2 独立入口和 B3 工具请求检查，再补正常与异常样例。不要改 B1 工具判断逻辑。

调整 Agent 决策：先判断是系统行为规则、B1 阶段流程还是 B4 协议问题。业务工具选择通常应由提示词和工具说明解决，不应在 B4 用关键词强制改写。

增加文件格式：读取能力放 file_reader 或独立 Skill；生成能力放 writer；路径和下载仍走 B3/后端的安全链路。

调整记忆策略：只修改 B5 和 memory 配置。B1 最多透传上下文包，不应实现召回算法。

调整模块展示页：优先拆纯展示组件和样例数据，保持后端接口、字段和观察/演示语义不变。

修改公共消息字段：必须同时核对 common schemas、B1、B3、B4、后端事件和前端类型。这类改动影响面最大，不能只修一处让页面暂时能显示。


十五、如何根据产物排查问题

模型没有调用工具：先看本次 B4 的 prompt_messages、raw_model_output 和 ai_message。确认模型是否看到了工具说明、是否输出了空工具列表，再检查系统提示和工具描述。

工具调用了但没有结果：看 tool_call_log 和 tool_messages。参数缺失、未知工具、路径错误和 Skill 异常都应在这里出现。

文件实际生成但前端没有下载：看 SkillResult 的 artifacts、relative_output_path 和 download_url，再检查后端 artifact 路径限制及前端消息适配。

最终回答与工具证据不一致：看 workspace 的 accepted_evidence、rejected_evidence、observations 和 answering 阶段输入，判断 observation 是否误判。

长对话忘记早期内容：看 workspace_memory_context、layered_memory_context 和 SQLite 召回日志，依次确认摘要是否存在、候选是否命中、向量/重排是否降级、来源消息是否加载。

回答结束后 B5 页面暂时为空：记忆反思在后台执行，先等待并刷新。若持续为空，再看 trace 中 turn_memory 状态和 memory_log。

停止后无法恢复：检查 checkpoints 下对应会话文件、前端消息的 resumable 状态和后端 resume 接口。恢复能力只适用于生成过有效检查点的 workspace 运行。


十六、协作红线

1. B1-B5 的边界优先于临时修复速度。工具逻辑不能塞进 B1，业务关键词判断不能塞进 B4，记忆召回不能放到前端。
2. 不回退无法确认来源的改动。仓库是多人协作状态，修改前先看 Git diff 和成员日志。
3. 运行产物、配置和源码要分清。outputs、memory 数据库和 conversation_prompts 多为运行数据，不应随意当作源码重写。
4. 校方材料保存在 B方向目录，只用于对照。
5. 项目 README 只维护根目录 README.md；本文是内部学习教程，不替代 README。
6. 任何“已完成”都要有代码路径或运行产物支撑。模型偶然成功一次不能代表跨模型稳定性，静态存在函数也不能代替实际验证。
7. 涉及模型、工具、记忆协议的改动必须说明风险、影响模块和未验证项。


十七、新成员建议学习顺序

第一阶段：理解概念
阅读本文第一至三章，再看 README 的项目概述和整体流程。能够口头说明 B1-B5 的边界，以及 AIMessage、ToolMessage、Skill 和 Tool Schema 的区别。

第二阶段：顺着一次真实任务读代码
从 backend/run_service.py 找到 B1 入口，再看 b1_agent_runtime.py、workspace 结构、B4 调用、B3 执行和 B2 Skill。最后看本次 outputs 中的 messages、trace 和工具日志，把代码和产物对应起来。

第三阶段：理解长对话
阅读 conversation_store 的表结构，再看 B5 reflection、retrieval、vector_retrieval 和 rerank。重点理解“摘要定位、来源作证”和“最近四轮保留原文”。

第四阶段：理解界面
先看 App 如何接收流式事件，再看某一个模块页如何调用后端接口。不要同时阅读五个大型模块页。

第五阶段：做一个小改动
优先选择单模块、低风险任务，例如补一个 B2 样例、拆一个纯展示组件或改一条准确的工具描述。修改前写清输入、输出和影响边界，修改后根据对应产物检查，不跨模块顺手重构。


十八、当前仍需警惕的事项

1. 默认 qwen_api 依赖网络、API 配额和本地代理，服务状态会影响 B4 和向量召回。
2. 本地 Qwen3.5-4B 权重不在仓库中，本地模式还依赖合适的 PyTorch 和硬件环境。
3. Python 沙箱是进程级限制，不是强安全隔离。
4. B1 批量任务、严格断点恢复报告、schema 注入对照、多模型统计、记忆冲突合并和错误记忆实验仍是明确缺口。
5. 旧版 memory 文档和 SQLite 主线的作用不同，讲解时不要互相代替。
6. 模型输出具有波动。现场判断必须以当次 raw output、标准消息和工具日志为准。
7. 一键启动脚本仍有临时连接配置，应继续按环境变量和本地配置管理，不能作为公开凭据来源。


结语

这个项目的核心不是工具数量，也不是页面数量，而是五个模块通过稳定数据结构完成闭环：B1 维护状态，B4 做模型决策，B3 管工具协议，B2 提供真实能力，B5 管理跨轮上下文。后续维护首先要保证这条链路和模块边界不被破坏，再考虑功能扩展和界面优化。
